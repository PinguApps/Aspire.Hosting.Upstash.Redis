## Purpose
Put the repository on the testing path you explicitly want: every test written in Reqnroll feature-file and step-definition style. This task exists to create the BDD-style test structure, shared helpers, and scenario map that all future implementation tasks will extend.

## Full Description
Set up the test project so every future task can add meaningful coverage without inventing its own test scaffolding. This task is not just about wiring Reqnroll packages; it must create a durable test architecture for the whole package.

The work must include:

1. Define the top-level feature grouping strategy.
   - API-shape features
   - local-behavior features
   - ownership-mode features
   - provider-client features
   - deploy/reconcile/output features
   - docs/sample validation features where practical
2. Create shared step infrastructure.
   - object/context model for scenario state
   - helper abstractions for fake Upstash responses
   - helper abstractions for Aspire model inspection
   - reusable assertions and builders
3. Decide how tests will simulate Upstash without hitting the real provider by default.
4. Decide how tests will inspect generated connection strings, outputs, annotations, and validation failures.
5. Create the initial feature files and step definitions needed to support later tasks cleanly.

This task should aim for a test suite that is:

- readable
- deterministic
- fast
- capable of covering both model-time and deploy-time behavior
- easy for future contributors to extend

The task must avoid two bad outcomes:

- a thin Reqnroll setup that later tasks have to fight
- a giant speculative test harness that is more complex than the product

Build only the infrastructure needed to support the current plan, but build it properly so later tasks can keep adding scenarios instead of rewriting the harness.

If live-provider tests appear useful, this task should define the shared pattern for opt-in live scenarios behind `UPSTASH_EMAIL` and `UPSTASH_API_KEY` plus any needed tags. That pattern must not make the default suite flaky or credential-dependent, and it must include teardown or restore hooks so live scenarios leave the Upstash account in the same state they found it.

Repository state rule: before the PR for this task is created, update any affected files in `plans/` including this file and any other plan files so they reflect the new repo state accurately. Keep `AGENTS.md` and `README.md` up to date in the same PR whenever the task changes behavior, workflow, structure, constraints, usage, or any other documented repo state. No PR from this task is complete until the implementation, tests, `README.md`, `AGENTS.md`, and all relevant plan files describe the same current reality.

## Expected Deliverables
- Reqnroll feature-file structure committed under the existing test project.
- Shared step definitions and scenario helpers that later tasks can extend.
- A written scenario map, either in the PR or test docs, showing where each planned behavior will be covered.
- Passing baseline scenarios proving the test harness itself works.

## Testing Requirements
- The task itself must introduce passing Reqnroll scenarios that prove:
  - the harness runs successfully
  - scenario state can be shared and asserted
  - fake provider interactions can be modeled deterministically
- If this task introduces live-test scaffolding, prove it skips cleanly without credentials and has a single shared cleanup path.
- Run the test project and the solution build.
- Do not leave skipped or broken scaffolding as the main output unless there is a documented and justified reason.

## PR Requirements
The PR must:

- Explain the test taxonomy and why it is structured that way.
- Show how future tasks are expected to add scenarios.
- Call out any intentionally deferred live-test work.

## Plan Maintenance Requirements
If the chosen test structure changes the scope of later test-heavy tasks:

- Update those plan files in the same PR.
- Keep the future tasks aligned with the actual harness now present in the repo.

## Done When
- Every later task has a clear place to add tests.
- The repo has a real Reqnroll-based testing foundation rather than only package references.
- The harness is passing and ready for incremental growth in every subsequent PR.
