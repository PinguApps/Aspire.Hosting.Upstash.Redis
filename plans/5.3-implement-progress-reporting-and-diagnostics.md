## Purpose
Make the deployment experience understandable and supportable. This task exists so `aspire deploy` shows meaningful progress and failures instead of opaque provider calls or stack traces.

## Full Description
Implement the logging, progress reporting, and diagnostic behavior for the package's publish/deploy integration. This should use the Aspire hook model chosen in `0.1` and present a clean operator experience during deployment.

The deployment experience should include clear phases such as:

- resolving configuration
- locating existing database
- creating database
- reconciling mutable settings
- validating immutable drift
- retrieving outputs

Each phase should report enough detail to be useful without leaking secrets. Failures must be phrased in language that an AppHost developer can act on quickly.

This task must also define the package's diagnostic standards:

- what is logged at normal verbosity
- what is omitted or redacted
- what additional detail is safe to include in thrown exceptions
- how provider identifiers are shown

The implementation should give special attention to secret redaction for:

- Upstash API key
- Redis password/token
- full connection strings where unnecessary
