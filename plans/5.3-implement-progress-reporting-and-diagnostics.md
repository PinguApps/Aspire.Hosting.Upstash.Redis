## Purpose
Make the deployment experience understandable and supportable. This task exists so `aspire deploy` shows meaningful progress and failures instead of opaque provider calls or stack traces.

## Current Baseline From `0.1`
Progress reporting should be implemented inside the Aspire `13.4.2` pipeline step registered by `PublishToUpstash(...)`. Use pipeline step activity/logging APIs rather than older publish/deploy callback annotations.

## Full Description
Implement the logging, progress reporting, and diagnostic behavior for the package's publish/deploy integration. This should use the Aspire hook model chosen in `0.1` and present a clean operator experience during deployment.

The deployment experience should include clear phases such as:

- resolving configuration
- locating existing database
- creating database
- reconciling mutable settings
- validating immutable drift
- retrieving outputs

Each phase should report enough detail to be useful without leaking secrets. Failures must be phrased in language that an AppHost developer can act on quickly.

This task must also define the package's diagnostic standards:

- what is logged at normal verbosity
- what is omitted or redacted
- what additional detail is safe to include in thrown exceptions
- how provider identifiers are shown

The implementation should give special attention to secret redaction for:

- Upstash API key
- Redis password/token
- full connection strings where unnecessary

## Implementation Notes
- Completed in task `5.3`.
- The deploy pipeline now reports these structured phases through an internal progress reporter that the Aspire pipeline step adapts to `PipelineStepContext.Logger`:
  - resolving configuration
  - locating the configured database by name
  - validating immutable drift before adoption/mutation
  - creating the database when ownership selects create
  - reconciling explicit mutable settings
  - retrieving Redis connection outputs from the final provider detail response
- Normal progress messages include the Redis resource name, explicit Upstash database name, and provider database id when known. Provider ids are treated as safe diagnostic identifiers.
- Diagnostics redact the Upstash Management API key, Redis password/token values, and full `redis://`/`rediss://` connection strings. Progress messages are created through the shared diagnostic helper so secrets are removed before they reach logs or test-visible diagnostics.
- Failure wording remains actionable at the point of failure: ownership failures name the ownership mismatch, immutable drift failures name the create-time-only setting, reconciliation failures name the mutable setting, and provider-contract output failures name the missing/invalid Redis connection field.
- Reqnroll coverage lives in `tests/Aspire.Hosting.Upstash.Redis/Features/DeployReconcileOutputs/DeploymentDiagnostics.feature`.
