# Implement Reconcile Flow for Mutable Settings

Step: `4.2`  
Stage: `4`  
Depends on: `0.2`, `2.2`, `2.3`, `3.1`, `3.2`, `3.3`  
Can run in parallel with: `4.1`, `4.3`  
Blocks: `5.1`, `5.2`, `6.3`

## Purpose
Make repeated deployments safe and useful by reconciling only the settings the caller explicitly set and that Upstash can safely mutate.

## Full Description
Implement the reconcile path for existing databases. This task must compare the caller's explicit desired settings with the actual provider state and apply only the supported, safe mutations required to bring the database into compliance.

The reconcile logic must:

1. Operate only on explicitly set values.
2. Leave unspecified values alone.
3. Re-fetch provider state as needed before and after mutation steps.
4. Apply updates in a deterministic order.
5. Surface actionable failures if a mutation cannot be performed.

Based on the capability matrix, this task should cover the mutable settings selected for v1, which are expected to include:

- read regions
- plan
- budget
- eviction

The `0.2` capability matrix excludes TLS from mutable reconciliation for v1. TLS is readable provider state and must be enabled; a requested or observed disabled TLS state belongs in the fail-fast path handled by `4.3`.

Each mutation path should be implemented deliberately rather than hidden behind a generic "patch everything" helper. The reason is clarity, testability, and accurate error reporting.

Use this deterministic update order unless implementation reveals a provider-specific reason to change it: read regions, plan, budget, eviction. Re-fetch provider state after each mutation class or after the full sequence if tests prove provider responses converge reliably; either way, final state must be verified before outputs are generated.

The task must also define how no-op reconciliation works. If the existing remote database already matches the explicit desired state, deployment should continue cleanly without unnecessary provider calls.

Where a setting is technically mutable but operationally risky, only implement it if the `0.2` investigation marked it as safe for v1. Otherwise, route that case into the immutable-drift/fail-fast path handled by `4.3`.

- Deterministic update ordering and no-op detection.
- Clear separation between safe mutable paths and fail-fast unsupported paths.

  - no-op reconcile when explicit settings already match
