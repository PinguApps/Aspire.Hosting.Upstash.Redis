## Decision Summary
This task is implemented by the decision record in [`plans/0.1-aspire-integration-decision.md`](0.1-aspire-integration-decision.md) and the initial skeleton in `src/Aspire.Hosting.Upstash.Redis/`.

The selected shape is:

- extend the existing `IResourceBuilder<RedisResource>` returned from `builder.AddRedis("cache")`
- attach `UpstashRedisDeploymentAnnotation` directly to the built-in `RedisResource`
- register deploy-time behavior through Aspire 13.4 `PipelineStepAnnotation` via `WithPipelineStepFactory`, with the Upstash step depending on `WellKnownPipelineSteps.DeployPrereq` and being required by `WellKnownPipelineSteps.Deploy`
- keep model-time work limited to public API calls, metadata capture, and pipeline-step registration
- leave Upstash API calls, parameter resolution, remote lookup, reconciliation, and output substitution for deploy-time implementation tasks

The code added in this task is intentionally only an implementation-ready skeleton. The registered pipeline step is a no-op until later deploy tasks replace the body.

## Purpose
Work out the exact Aspire 13.4.2 integration mechanism that this package should use so the implementation is built on the real hosting and deployment extension points instead of assumptions. This task is intentionally investigative first. It should leave the repository with an unambiguous technical decision about how an `AddRedis("cache")` resource can be annotated for Upstash deployment without breaking the normal Aspire local-development experience.

## Full Description
Investigate Aspire 13.4.2 in detail and answer the following questions with evidence from source, docs, and small proof-of-concept code where needed:

1. Whether the package should extend the built-in `RedisResource` directly or introduce a wrapper/custom resource that composes with it.
2. Whether the package can safely attach custom annotations to the existing Redis resource builder returned from `builder.AddRedis(...)`.
3. Which publish/deploy hooks are correct for this scenario:
   - `DeployingCallbackAnnotation` only
   - `PublishingCallbackAnnotation` plus `DeployingCallbackAnnotation`
   - another supported Aspire mechanism discovered during the investigation
4. How Aspire represents Redis connection strings for built-in Redis resources and what must remain true so `WithReference(cache)` keeps working after this package is added.
5. How to avoid changing any `aspire run` or local dashboard behavior when `.PublishToUpstash(...)` is present.
6. Whether Aspire provides a supported way to persist per-resource deployment state across deploys, or whether deterministic re-lookup by explicit remote name is the correct fallback.
7. What internal annotations or resource metadata need to exist so later tasks can resolve:
   - explicit remote database name
   - ownership mode
   - management credentials
   - option values and explicit-setting tracking
