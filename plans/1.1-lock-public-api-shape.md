# Lock Public API Shape

Step: `1.1`  
Stage: `1`  
Depends on: `0.1`, `0.2`

Blocks: `2.1`, `2.3`, `3.1`, `3.2`, `5.1`, `7.1`, `7.2`

## Purpose
Define and implement the public package API that consumers will use in their AppHost code. This task exists to turn the agreed product contract into a stable, ergonomic, Aspire-native fluent API before the deeper deployment logic is built.

## Full Description
Design the consumer experience for the package and commit the exact public surface. The design must satisfy these already-locked rules:

- The consumer starts from standard Aspire Redis: `builder.AddRedis("cache")`.
- Upstash behavior is opt-in via a single publish-oriented method.
- The method must support create-only, existing-only, and create-or-adopt semantics.
- Remote database name must be explicit and required.
- Upstash management auth must be modeled as separate email and API key inputs.
- The API must support Aspire parameters and other safe value sources rather than forcing hard-coded strings.

This task must settle and implement:

1. Extension method naming.
2. Overload strategy.
3. Ownership mode shape.
4. Options object shape.
5. Required values vs optional values.
6. How explicit-setting tracking is represented internally without making the public API awkward.
7. XML documentation comments for all public members.
8. Nullability and argument validation behavior.

The design should strongly prefer clarity over cleverness. A likely direction is one fluent entry point that takes the always-required values first and leaves optional values in a builder/options callback, but this task is responsible for choosing the best final shape. The final result must make common usage obvious while still allowing advanced configuration without sprawling overloads.

The API must also make ownership intent extremely clear at the call site. A developer reading the AppHost should be able to tell whether the deployment is supposed to create, require an existing database, or create-or-adopt on collision.

When designing the public shape, account for:

- Aspire `ParameterResource` use
- plain strings sourced from configuration
- secret-vs-non-secret values
- documentation readability
- future extensibility without forcing speculative abstractions into v1

If this task concludes that the previously discussed sample shape is not good enough, it must replace it with a stronger shape and explain why in the PR. The consumer experience is the main output of this task.

Repository state rule: before the PR for this task is created, update any affected files in `plans/` including this file and any other plan files so they reflect the new repo state accurately. Keep `AGENTS.md` and `README.md` up to date in the same PR whenever the task changes behavior, workflow, structure, constraints, usage, or any other documented repo state. No PR from this task is complete until the implementation, tests, `README.md`, `AGENTS.md`, and all relevant plan files describe the same current reality.

  - create-only
  - existing-only
  - create-or-adopt

- Add tests that verify parameter-based inputs can be represented without leaking secrets into the wrong place.
