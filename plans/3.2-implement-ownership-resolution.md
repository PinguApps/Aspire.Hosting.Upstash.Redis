## Purpose
Encode the central provisioning behavior of the package: whether deployment must create a database, must find an existing one, or may create-or-adopt. This task exists to make the contract explicit and deterministic before the create/reconcile implementation lands.

## Full Description
Implement the ownership-mode decision logic using the final public API and the Upstash client capabilities already established. The behavior must exactly match the product requirements:

- create-only: fail if a matching database already exists when creation is required to be exclusive
- existing-only: fail if the named database does not already exist
- create-or-adopt: use an existing compatible database if found, otherwise create one

This task must define the authoritative algorithm for deciding which path to take from a resolved deployment request. It should not yet perform all create/reconcile details, but it must establish:

- how remote identity is looked up
- how collisions are interpreted
- which path each ownership mode chooses
- what high-level errors are raised when the selected path is impossible

The logic must be safe for repeated deployments and explicit enough that later tasks can layer concrete create and reconcile steps onto it without changing semantics.

This task is also where user-facing failure wording for ownership mismatches should be defined. The failure messages should make it obvious whether the problem was:

- existing database found when create-only was requested
- database missing when existing-only was requested
- existing database incompatible with requested explicit settings

## Implementation Result
Task `3.2` implemented the internal ownership-resolution decision engine in `src/Aspire.Hosting.Upstash.Redis/Deployment/`.

The authoritative algorithm is:

1. Resolve the deploy-time request to an explicit database name, ownership mode, and provider deployment options.
2. Look up remote identity by calling `IUpstashRedisManagementClient.FindDatabaseByNameAsync(databaseName, cancellationToken)`. The management client performs list-by-account, requires at most one exact database-name match, and fetches credential-bearing details for the match.
3. Branch by ownership mode:
   - `CreateOnly`: choose the create path only when no existing database is found; fail with `CreateOnlyDatabaseAlreadyExists` when the name already exists.
   - `ExistingOnly`: choose the adopt path only when a compatible existing database is found; fail with `ExistingOnlyDatabaseMissing` when the name is absent.
   - `CreateOrAdopt`: choose adopt when a compatible existing database is found; otherwise choose create.
4. Before adopting, reject incompatible existing databases for immutable/read-only settings the resolver can verify from current provider details. This task covers explicit primary-region checks and rejects provider state with TLS disabled; mutable settings remain for `4.2`, and broader immutable drift remains for `4.3`.

The stable user-facing failure reasons are represented by `UpstashRedisOwnershipResolutionFailureReason` and surfaced through `UpstashRedisOwnershipResolutionException`:

- `CreateOnlyDatabaseAlreadyExists`: the requested create-only database name already exists.
- `ExistingOnlyDatabaseMissing`: the requested existing-only database name does not exist.
- `ExistingDatabaseIncompatible`: an existing database was found but conflicts with requested explicit settings or required provider state that must not be silently replaced or mutated in v1.

Reqnroll coverage was added in `tests/Aspire.Hosting.Upstash.Redis/Features/OwnershipModes/OwnershipResolution.feature` using a deterministic fake management client. The scenarios verify create-only success/collision failure, existing-only adopt/missing failure, create-or-adopt create/adopt paths, lookup-by-name behavior, and failure message wording.
