# Implement Ownership Resolution

Step: `3.2`  
Stage: `3`  
Depends on: `1.1`, `1.2`, `2.2`, `2.3`  
Can run in parallel with: `3.1`, `3.3`  
Blocks: `4.1`, `4.2`, `4.3`, `6.2`

## Purpose
Encode the central provisioning behavior of the package: whether deployment must create a database, must find an existing one, or may create-or-adopt. This task exists to make the contract explicit and deterministic before the create/reconcile implementation lands.

## Full Description
Implement the ownership-mode decision logic using the final public API and the Upstash client capabilities already established. The behavior must exactly match the product requirements:

- create-only: fail if a matching database already exists when creation is required to be exclusive
- existing-only: fail if the named database does not already exist
- create-or-adopt: use an existing compatible database if found, otherwise create one

This task must define the authoritative algorithm for deciding which path to take from a resolved deployment request. It should not yet perform all create/reconcile details, but it must establish:

- how remote identity is looked up
- how collisions are interpreted
- which path each ownership mode chooses
- what high-level errors are raised when the selected path is impossible

The logic must be safe for repeated deployments and explicit enough that later tasks can layer concrete create and reconcile steps onto it without changing semantics.

This task is also where user-facing failure wording for ownership mismatches should be defined. The failure messages should make it obvious whether the problem was:

- existing database found when create-only was requested
- database missing when existing-only was requested
- existing database incompatible with requested explicit settings

## Expected Deliverables
- Ownership-resolution logic and supporting types/helpers.
- Clear branching behavior for all three modes.
- Stable, human-readable failure reasons that later deploy tasks can reuse.

## Testing Requirements
- Add Reqnroll scenarios that verify:
  - create-only success and collision failure
  - existing-only success and missing-resource failure
  - create-or-adopt create path and adopt path
  - failure messages identify the ownership mismatch clearly
- Use deterministic provider fakes rather than live Upstash.
- Run the test suite and `dotnet build ./Aspire.Hosting.Upstash.Redis.slnx`.

## PR Requirements
The PR must:

- Describe the ownership algorithm in detail.
- Show the exact branching behavior for each mode.
- Highlight any edge cases intentionally excluded from v1.

## Plan Maintenance Requirements

- Update `4.1`, `4.2`, `4.3`, and `6.2` accordingly in the same PR.

## Done When

- Later tasks can build on a stable decision engine instead of redefining behavior.
