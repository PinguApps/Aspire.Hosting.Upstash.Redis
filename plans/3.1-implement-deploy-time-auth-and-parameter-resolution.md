# Implement Deploy-Time Auth and Parameter Resolution

Step: `3.1`  
Stage: `3`  
Depends on: `0.1`, `1.1`, `1.2`, `2.1`, `2.3`  
Can run in parallel with: `3.2`, `3.3`  
Blocks: `4.1`, `4.2`, `4.3`, `5.1`, `5.2`, `5.3`

## Purpose
Resolve the values needed for Upstash deployment at the correct time and with the correct secret boundaries. This task exists to make sure management credentials and remote names are handled through Aspire-native parameter patterns instead of leaking into application outputs or local runtime behavior.

## Full Description
Implement the mechanism that turns the resource's configured inputs into concrete deploy-time values when `aspire deploy` runs. This task must respect the separation between:

- local AppHost composition
- publish/deploy parameter prompting and caching
- application-facing Redis outputs
- infrastructure-only Upstash management credentials

The task must support the API shape finalized in `1.1`, including parameter-based usage for:

- Upstash account email
- Upstash API key
- explicit remote database name
- any optional settings that may also be parameterized

The implementation must:

1. Resolve values only in the deployment path selected by `0.1`.
2. Keep the management API key out of application outputs and references.
3. Produce clear failures for missing required deploy-time values.
4. Preserve Aspire-native prompting/caching behavior where parameters are unresolved.
5. Ensure local `aspire run` does not require Upstash credentials when only local Redis behavior is used.

This task should also document the package's preferred secret story for consumers:

- parameter resources for secrets
- safe configuration fallback patterns, if supported
- no hard-coded secrets in code samples

## Expected Deliverables
- Deploy-time resolution logic for management email, API key, remote name, and relevant option values.
- Clear separation between infrastructure-only values and app-facing values.
- Internal helpers that later tasks can call without re-implementing resolution.

## Testing Requirements
- Add Reqnroll scenarios that verify:
  - required deploy-time values are resolved correctly
  - missing values fail with actionable errors
  - secret values do not leak into app-facing outputs
  - local model construction does not force deploy-only values during normal local use
- Run the test suite and `dotnet build ./Aspire.Hosting.Upstash.Redis.slnx`.

## PR Requirements
The PR must:

- Explain when and how values are resolved.
- Prove that the management API key is not exposed to consuming applications.
- Call out any supported configuration-source patterns beyond plain parameters.

## Plan Maintenance Requirements
If Aspire parameter resolution behaves differently than expected:

- Update the affected plan files immediately.
- Keep code, docs, and task descriptions aligned.

## Done When
- Later deploy tasks can obtain concrete, validated Upstash inputs safely.
- Secret boundaries are enforced and covered by passing Reqnroll scenarios.
