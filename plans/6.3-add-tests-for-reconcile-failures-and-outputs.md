## Purpose
Close the hardest correctness gaps around repeated deploy safety, failure quality, and the app-facing output contract.

## Full Description
Expand the Reqnroll suite to cover the full reconcile and output surface in depth. This is the hardening pass for provider-state correctness and app-facing output fidelity.

The scenarios must deepen or complete coverage for:

- no-op reconcile and each supported mutable reconciliation path, building on the baseline `4.2` scenarios
- immutable drift failures
- unsupported mutation failures and additional provider failure classes not already covered by `4.2`
- output generation for the connection string
- output generation for host, port, password, TLS, and database name
- secret redaction in diagnostics
- missing deploy-time values where relevant to the output path

Add explicit coverage that TLS is not reconciled as mutable, `reset-password` is never called automatically, and missing `password` in a credential-bearing provider detail response fails clearly.

This task should also add the opt-in live-provider scenarios that are most valuable for real Upstash validation of deploy, repeat-deploy, and output behavior. Gate them explicitly on `UPSTASH_EMAIL` and `UPSTASH_API_KEY`, document the gating clearly, and keep the default suite deterministic. Any live reconcile scenario must either use a disposable database that is deleted in teardown or snapshot and restore every mutable setting it changes before exit so the account is left unchanged after the run.
