## Purpose
Enforce the safety rule that if the existing remote database conflicts with an explicitly requested setting that cannot be reconciled safely, deployment must fail clearly instead of silently drifting.

## Full Description
Implement the fail-fast path for immutable or unsafe drift. This task is the guardrail that keeps the package predictable in production. It must compare explicit requested values with provider state and detect mismatches that should not be auto-corrected in v1.

Task `3.2` already rejects incompatible existing databases for explicit primary-region and TLS values while resolving ownership. This task must extend that guardrail into the final immutable-drift set and avoid redefining ownership branching.

The task must define and enforce the final immutable-drift set, expected to include items such as:

- platform, where detectable
- primary region, building on the `3.2` resolver check
- TLS disabled or explicitly requested as disabled, building on the `3.2` resolver check
- database name identity mismatch
- any setting found in `0.2` to be unsafe or unsupported to mutate
- any identity-level mismatch that would target the wrong remote resource

For each detected drift case, the package must:

1. Stop deployment before destructive or misleading behavior occurs.
2. Produce a highly actionable error message.
3. Tell the user what was requested, what exists remotely, and why the package refuses to auto-fix it.

The package should not try to replace or rename remote resources automatically in v1. This task must enforce that boundary and make the failure experience good enough that users understand the remediation path.

The provider has rename, delete, reset-password, and enable-TLS endpoints, but the `0.2` matrix deliberately excludes them from automatic v1 drift repair. Existing database mismatches for create-time-only or unsupported settings must fail before any mutation is attempted.

## Implemented Immutable/Unsafe Set
- Database name identity mismatch: fails because the configured database name is the v1 remote identity and the package must not rename or adopt another resource automatically.
- Platform mismatch where detectable: fails when the existing primary region maps to a different platform than the explicit request because platform is create-time-only and has no safe update endpoint.
- Primary region mismatch: fails for explicit requested primary region drift because primary region is create-time-only and the package must not move or replace the database automatically.
- TLS disabled state: fails even when TLS was not explicit because TLS is required-on/read-only in v1 and the package must not call TLS repair endpoints automatically.
- Mutable settings, currently read regions, plan, budget, and eviction, are intentionally excluded from immutable drift detection and remain owned by mutable reconcile work.
