# Implement Package Resource Annotations and State Model

Step: `2.1`  
Stage: `2`  
Depends on: `1.1`, `1.2`  
Can run in parallel with: `2.2`, `2.3`  
Blocks: `3.1`, `3.2`, `3.3`, `5.1`, `5.2`, `5.3`

## Purpose
Create the internal resource metadata model that hangs Upstash deployment intent off a standard Aspire Redis resource. This is the foundation that lets later tasks resolve configuration and execute deploy-time behavior without changing the normal local Redis experience.

## Full Description
Implement the internal representation of "this Redis resource should publish to Upstash" in a way that is faithful to the `0.1` investigation result. The implementation must capture all package-specific intent at model-build time and make it available at deploy time.

The work must include:

1. Attach package-specific annotations or metadata to the Redis resource builder selected in `0.1`.
2. Represent all Upstash-specific settings in a single internal state model that later tasks can consume.
3. Preserve explicit-setting information so reconcile logic can tell the difference between:
   - a caller explicitly setting a value
   - a default chosen by the API surface
   - a value left unspecified and therefore not to be reconciled
4. Ensure the extension does not alter local Redis container behavior, connection references, or dashboard expectations during `aspire run`.
5. Make the internal state easy to inspect in tests without exposing unnecessary surface publicly.

This task should not yet perform Upstash network calls. It should only create the reliable model-time data structures and resource annotations that deploy-time tasks will later consume.

Particular care is required around the boundary between public and internal types:

- Public types must remain simple and consumer-friendly.
- Internal state must be precise enough for deployment logic.
- Sensitive values must be represented as references/parameters where appropriate, not flattened into logs or unsafe strings.

## Expected Deliverables
- Internal annotation/state model committed in `src/`.
- Resource builder extension wiring that stores the package state against the chosen Aspire resource.
- No regression to baseline Redis reference behavior.

## Expected Areas
- Internal annotation classes
- Internal state/definition classes
- Public extension method backing logic
- Any resource-builder helpers needed to retrieve the attached state

## Testing Requirements
- Add Reqnroll scenarios that verify:
  - calling the public API attaches the Upstash state to the Redis resource
  - explicit vs implicit option values are distinguishable
  - local Redis behavior remains unchanged when Upstash publishing is configured
  - the standard reference chain still exists for dependent projects
- Run the relevant tests and `dotnet build ./Aspire.Hosting.Upstash.Redis.slnx`.

## PR Requirements
The PR must:

- Explain the internal state shape and why it is sufficient for later deploy tasks.
- Call out how explicit-setting tracking works.
- Prove that this task still does not introduce remote Upstash behavior into local runs.

## Plan Maintenance Requirements
If implementing the state model reveals missing fields or flawed downstream assumptions:

- Update the affected plan files in the same PR.
- Keep the internal state and the documented downstream expectations aligned.

## Done When
- Later deploy tasks can recover a complete, precise Upstash intent model from the Redis resource.
- Tests prove the annotation/state model is correct and non-invasive to local Aspire behavior.
