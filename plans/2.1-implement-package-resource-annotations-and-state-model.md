## Purpose
Create the internal resource metadata model that hangs Upstash deployment intent off a standard Aspire Redis resource. This is the foundation that lets later tasks resolve configuration and execute deploy-time behavior without changing the normal local Redis experience.

## Current Baseline From `0.1`
Task `0.1` added the first model-time skeleton:

- `UpstashRedisDeploymentAnnotation` is attached directly to Aspire's built-in `RedisResource`.
- `UpstashRedisDeploymentOptions` tracks explicitly set option names internally.
- `UpstashRedisValue` captures literal-or-parameter value sources for required inputs and optional string settings.
- `PublishToUpstash(...)` registers a no-op deploy pipeline step.

This task should harden that skeleton into the final internal state model after `1.1` locks the public API. It should not revisit wrapper/custom resource alternatives unless Aspire behavior has changed.

## Full Description
Implement the internal representation of "this Redis resource should publish to Upstash" in a way that is faithful to the `0.1` investigation result. The implementation must capture all package-specific intent at model-build time and make it available at deploy time.

The work must include:

1. Attach package-specific annotations or metadata to the Redis resource builder selected in `0.1`.
2. Represent all Upstash-specific settings in a single internal state model that later tasks can consume.
3. Preserve explicit-setting information so reconcile logic can tell the difference between:
   - a caller explicitly setting a value
   - a default chosen by the API surface
   - a value left unspecified and therefore not to be reconciled
4. Ensure the extension does not alter local Redis container behavior, connection references, or dashboard expectations during `aspire run`.
5. Make the internal state easy to inspect in tests without exposing unnecessary surface publicly.

This task should not yet perform Upstash network calls. It should only create the reliable model-time data structures and resource annotations that deploy-time tasks will later consume.

Particular care is required around the boundary between public and internal types:

- Public types must remain simple and consumer-friendly.
- Internal state must be precise enough for deployment logic.
- Sensitive values must be represented as `UpstashRedisValue` parameter references where appropriate, not flattened into logs or unsafe strings.

## Implementation Notes
- Complete in PIN-157.
- `PublishToUpstash(...)` attaches an internal `UpstashRedisDeploymentAnnotation` to Aspire's built-in `RedisResource`.
- The annotation owns a single internal `UpstashRedisDeploymentState` containing the remote database name, ownership mode, infrastructure-only account email/API key sources, and a snapshot of `UpstashRedisDeploymentOptions`.
- `RedisResource.GetUpstashRedisDeploymentState()` is the internal retrieval helper for later deploy tasks and tests.
- Option explicit-setting tracking is preserved in the options snapshot, including explicit `null` values, so later reconcile logic can distinguish configured intent from unspecified settings.
- The state snapshot is defensive: mutating the original callback options, explicit-setting set copies, or configured read-region collection does not mutate stored deployment intent.
- Reqnroll scenarios cover annotation/state attachment, parameter-backed sources, explicit versus unspecified option intent, unchanged Redis connection properties, and the standard reference chain.
