# Implement Remote Lookup and Stable Identity Handling

## Purpose
Make repeated deployments deterministic by defining how a configured Aspire resource finds and continues to use the same remote Upstash database over time. This task exists to close the gap between "explicit remote name is required" and "repeated deploys should keep targeting the original database safely."

## Current Baseline From `0.1`
Aspire `13.4.2` exposes `IDeploymentStateManager` for deployment-state sections, but the accepted v1 identity strategy remains deterministic lookup by explicit remote database name. Deployment state may cache provider identifiers only as an optimization or diagnostic aid; it must not become the only way to find the intended database.

## Full Description
Implement the lookup and identity strategy chosen from the earlier investigations. This task must establish the package's rules for how a Redis resource maps to a remote Upstash database across:

- first deployment
- repeated deployment
- explicit remote name changes
- out-of-band remote drift

The implementation must answer, in code:

1. Whether the package identifies the target primarily by explicit remote name, provider id, or a hybrid of both.
2. Whether any stable identifier is persisted or cached between deploys when Aspire makes that practical.
3. What happens when the explicit remote name now resolves to:
   - the same resource
   - a different resource
   - no resource
4. How the package prevents accidental takeover of the wrong database when names collide or external changes occur.

The safest v1 result is more important than convenience. If the framework or provider limits state persistence, this task should fall back to a deterministic, transparent lookup strategy and make the trade-off explicit in code comments, tests, and PR notes.

This task should also define how name changes are interpreted. Because the remote name is explicit and user-controlled, changing it should generally be treated as selecting a different remote resource identity rather than silently renaming the existing one. If a different behavior is chosen, this task must justify it carefully.

The `0.2` capability matrix confirms there is no documented get-by-name endpoint. Implement lookup through `GET /redis/databases` and require exactly one configured-name match. Provider id is the only unique provider identity; persist and prefer it when Aspire deployment state provides a safe mechanism, but verify the fetched database still has the configured explicit name. Never call the provider rename endpoint in v1.
