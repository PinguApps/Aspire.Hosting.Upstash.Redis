## Purpose
Make the deployed Upstash-backed Redis resource behave like a normal Aspire Redis dependency from the consuming application's point of view. This task exists to preserve the key product promise: `WithReference(cache)` should still work.

## Current Baseline From `0.1`
Aspire's built-in `RedisResource` remains the resource of record. Its standard connection surface is `IResourceWithConnectionString` with a connection string based on host/port, optional password, and TLS, plus connection properties named `Host`, `Port`, optional `Password`, and `Uri`. This task must preserve that surface rather than introducing a parallel wrapper resource.

## Full Description
Implement the app-facing Redis connection output for the Upstash-backed deployment path. The consuming application should receive a usable Redis connection string through the normal Aspire reference flow, with no need to know that the backing resource is Upstash.

This task must:

1. Determine the exact connection-string format Aspire's Redis ecosystem expects in practice.
2. Build that string from the resolved Upstash connection details.
3. Ensure the value is available through the same resource reference pattern used by standard Redis resources.
4. Keep the output secret-safe.
5. Avoid breaking the local-run connection behavior of the original Redis resource.

The implementation must explicitly verify that the generated connection string is compatible with the intended .NET Redis client expectations. If the built-in Aspire Redis integration includes quirks or conventions that must be mirrored, this task is responsible for mirroring them.

The `0.2` capability matrix requires connection details to come from a credential-bearing `GET /redis/database/{id}` response: `password`, `endpoint`, `port`, and `tls`. Do not use Upstash management credentials in app-facing outputs. If the provider returns an endpoint slug rather than a full host, normalize only according to a confirmed provider rule; otherwise fail clearly rather than generating a broken connection string.

The output logic must also account for the fact that the resource has two modes:

- local mode: normal Aspire Redis behavior
- deploy mode: Upstash-backed connection details

Do not implement these as two unrelated systems. The goal is one coherent resource experience with deploy-time provider substitution behind the scenes.
