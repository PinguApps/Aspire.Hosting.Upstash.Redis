# Run Full Validation and Release Hardening

Step: `7.3`  
Stage: `7`  
Depends on: `5.3`, `6.1`, `6.2`, `6.3`, `7.1`, `7.2`  
Can run in parallel with: `none`

## Purpose
Act as the final quality gate. This task exists to prove the package is coherent as a whole, that the docs match the implementation, and that the repository is ready for a release-quality PR state.

## Full Description
Run the complete validation pass after all earlier implementation, testing, and documentation tasks are merged. This task is not just "run tests." It is the final hardening review for the end-to-end product.

The work must include:

1. Run the full build and Reqnroll test suite.
2. Verify README and sample material still match the real public API.
3. Verify plan files are still accurate after the actual implementation work.
4. Fix any remaining inconsistencies, flaky tests, wording issues, or integration gaps discovered during the full pass.
5. Confirm that the package still satisfies the locked product contract:
   - local Redis remains normal during local use
   - Upstash work happens only in deploy flow
   - ownership modes behave correctly
   - repeated deploys are safe
   - immutable drift fails clearly
   - app-facing outputs are correct
   - management secrets stay out of app outputs

This task should also evaluate whether there is any remaining unplanned work that is genuinely required for "release quality." If yes:

- update `plans/` in the same PR
- do not hand-wave it as future polish

Repository state rule: before the PR for this task is created, update any affected files in `plans/` including this file and any other plan files so they reflect the new repo state accurately. Keep `AGENTS.md` and `README.md` up to date in the same PR whenever the task changes behavior, workflow, structure, constraints, usage, or any other documented repo state. No PR from this task is complete until the implementation, tests, `README.md`, `AGENTS.md`, and all relevant plan files describe the same current reality.

## Expected Deliverables
- A clean full build and passing test suite.
- Final bug fixes and consistency fixes needed to reach release quality.
- Final synchronization between implementation, docs, samples, and plans.

## Testing Requirements
- Run the full test suite.
- Run `dotnet build ./Aspire.Hosting.Upstash.Redis.slnx`.
- Run any additional validation commands introduced by the repo by this point.
- If opt-in live tests exist and `UPSTASH_EMAIL` plus `UPSTASH_API_KEY` are available, run them and report the result clearly. If they are not run, state that explicitly in the PR.
- Confirm the live-test run left no remote Upstash resources or settings behind beyond the pre-test baseline.

## PR Requirements
The PR must:

- Provide a thorough validation summary.
- Call out exactly which commands were run.
- State whether live-provider validation was run or not.
- State how live-provider validation verified cleanup or state restoration.
- Mention any last-minute plan updates made to keep `plans/` truthful.

## Plan Maintenance Requirements
If this final gate discovers missing release-critical work:

- Update the affected plan files or create new ones immediately before merge.

## Done When
- The repository is internally consistent and release-ready.
- There are no known undocumented mismatches between code, tests, docs, samples, and plans.
