# Implement Option and Domain Model

Step: `2.3`  
Stage: `2`  
Depends on: `0.2`, `1.1`, `1.2`  
Can run in parallel with: `2.1`, `2.2`  
Blocks: `3.1`, `3.2`, `4.1`, `4.2`, `4.3`, `7.1`

## Purpose
Encode the Upstash-facing domain vocabulary cleanly and safely. This task exists to convert raw strings and loosely related settings into a precise model for platform, region, plan, ownership mode, and related option values.

## Full Description
Implement the type system around the public and internal options so the package does not rely on magic strings or ad hoc validation later. This task should produce the reusable domain layer that the rest of the implementation builds on.

The domain model must include:

- ownership mode
- cloud platform/provider
- primary region
- read regions
- plan
- budget
- eviction
- TLS
- explicit remote database name

It must also define:

- how public values map to Upstash API values
- what validation occurs at configuration time vs deploy time
- how invalid combinations are rejected

The implementation should be pragmatic:

- use value objects where they add clarity
- use enums where the provider domain is fixed and well understood
- avoid speculative abstractions for future providers or non-Redis products

Special care is required for region modeling. The package should give consumers a strongly typed experience without forcing an awkward explosion of types. If a composite model like `new Region(provider, location)` is retained, this task must justify it. If a simpler model is better, this task should simplify decisively.

This task is also responsible for making sure the public model and the provider model remain decoupled enough that docs stay readable and provider serialization remains internal.

## Expected Deliverables
- Public enums/value types/options helpers where appropriate.
- Internal mapping logic from package domain values to Upstash request values.
- Validation helpers for required values and incompatible combinations.

## Testing Requirements
- Add Reqnroll scenarios that verify:
  - valid domain values map correctly to provider payload values
  - invalid combinations fail with clear messages
  - explicit-setting information is preserved correctly when the domain model is built
  - region, plan, and ownership values are all represented accurately
- Run the test suite and `dotnet build ./Aspire.Hosting.Upstash.Redis.slnx`.

## PR Requirements
The PR must:

- Explain the chosen domain-model shape, especially for regions and plans.
- Show how validation errors are phrased and where they occur.
- Confirm the model stays focused on Upstash Redis only.

## Plan Maintenance Requirements
If domain-model choices materially change API usage or later implementation:

- Update the corresponding plan files in the same PR.

## Done When
- Later tasks can consume typed, validated values instead of loose strings.
- The domain model is covered by passing Reqnroll scenarios and is stable enough for the rest of the implementation.
