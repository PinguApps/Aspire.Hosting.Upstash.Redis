# Implement Option and Domain Model

Step: `2.3`  
Stage: `2`  
Depends on: `0.2`, `1.1`, `1.2`  
Can run in parallel with: `2.1`, `2.2`  
Blocks: `3.1`, `3.2`, `4.1`, `4.2`, `4.3`, `7.1`

## Purpose
Encode the Upstash-facing domain vocabulary cleanly and safely. This task exists to convert raw strings and loosely related settings into a precise model for platform, region, plan, ownership mode, and related option values.

## Full Description
Implement the type system around the public and internal options so the package does not rely on magic strings or ad hoc validation later. This task should produce the reusable domain layer that the rest of the implementation builds on.

The domain model must include:

- ownership mode
- cloud platform/provider
- primary region
- read regions
- plan
- budget
- eviction
- TLS as required-on/read-only provider state, not a safely mutable setting
- explicit remote database name
- literal-or-parameter source handling already locked by `1.1` through `UpstashRedisValue`

It must also define:

- how public values map to Upstash API values
- what validation occurs at configuration time vs deploy time
- how invalid combinations are rejected

The `0.2` capability matrix settled these domain constraints:

- `platform` and `primary_region` are create-time-only.
- `read_regions`, `plan`, `budget`, and `eviction` preserve explicit-setting information for reconcile.
- Public option properties for string-like values currently use `UpstashRedisValue`; this task should add provider-domain validation/mapping behind that source model rather than redesigning the public value-source boundary unless a blocking flaw is found.
- TLS must be modeled as enabled for v1. If the public API exposes TLS at all, `false` must be rejected clearly; downstream reconciliation must not try to disable TLS.
- Database name is an explicit remote identity selector, not a mutable display name.
- `auto_upgrade`, `prod_pack`, backup, ACL, private networking, delete, rename, and password reset are outside the v1 domain model.

The implementation should be pragmatic:

- use value objects where they add clarity
- use enums where the provider domain is fixed and well understood
- avoid speculative abstractions for future providers or non-Redis products

Special care is required for region modeling. The package should give consumers a strongly typed experience without forcing an awkward explosion of types. If a composite model like `new Region(provider, location)` is retained, this task must justify it. If a simpler model is better, this task should simplify decisively.

This task is also responsible for making sure the public model and the provider model remain decoupled enough that docs stay readable and provider serialization remains internal.

## Implementation Result
State: `Complete`

Task `2.3` added the Upstash Redis option/domain model behind the locked `1.1` public value-source boundary.

Committed public domain helpers:

- `UpstashRedisCloudPlatform`
  - `Aws`
  - `Gcp`
- `UpstashRedisRegion`
  - provider-prefixed AWS and GCP global database regions, for example `AwsEuWest1`, `AwsEuWest2`, and `GcpEuropeWest1`
- `UpstashRedisPlan`
  - `Free`, `PayAsYouGo`, and the fixed-size plans currently accepted by the Upstash Redis management API
- `UpstashRedisDeploymentOptions` helper methods:
  - `SetPlatform(...)`
  - `SetPrimaryRegion(...)`
  - `SetReadRegions(...)`
  - `SetPlan(...)`
  - `SetBudget(...)`

The region model is a single provider-prefixed enum rather than a composite `Region(provider, location)` value. This keeps consumer code simple, avoids an explosion of provider-specific types, and still makes ambiguous or provider-specific region codes clear at the call site.

Internal provider-domain mapping now projects configured options to Upstash API payload values while preserving the original `UpstashRedisValue` source. Literal values are validated during AppHost model construction. Parameter-backed values retain their source with no literal provider value, so deploy-time parameter resolution can validate the resolved value later.

Configuration-time validation now rejects:

- unsupported literal platforms, regions, plans, and budgets
- disabled TLS
- primary regions whose provider does not match the configured platform
- read regions that duplicate each other, match the primary region, are unsupported as read regions, or cross provider/platform boundaries
- budgets configured with an explicit non-pay-as-you-go plan

The model remains focused on Upstash Redis only. `auto_upgrade`, `prod_pack`, backup, ACL, private networking, delete, rename, and password reset remain outside v1.

Reqnroll coverage was added to `Features/ApiShape/PublishToUpstash.feature` for provider payload mapping, invalid combinations, explicit-setting preservation, parameter-backed source preservation, region representation, plan representation, and ownership representation.
