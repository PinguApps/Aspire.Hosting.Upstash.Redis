# Implement Upstash Management Client Layer

Step: `2.2`

Depends on: `0.2`, `1.2`

## Purpose
Build the typed Upstash HTTP management client that all deploy-time behavior will use. This task exists to isolate provider communication, keep the deployment logic readable, and make provider interactions testable without live network access.

## Full Description
Implement a small, robust client layer for the Upstash Redis management API based strictly on the capability matrix from `0.2`. This client should be intentionally narrow: only the endpoints and data required by v1 should be implemented.

The client must cover, as confirmed by the investigation:

- authentication header construction
- list or lookup operations needed to find a database by name and/or id
- database creation
- database detail retrieval, including credential fields needed for app outputs
- mutable operations required by the v1 config surface
- any needed state/readiness polling

The client design should:

- use typed request and response models
- use predictable serialization settings
- support cancellation
- surface provider failures in a way later tasks can translate into clear deploy errors
- be easy to fake in tests

Do not bury deployment-policy decisions in this layer. The client should express provider capabilities, not ownership semantics. For example:

- "find database by name" belongs here
- "create-or-adopt if not found" does not

Likewise, the client should not decide what is mutable vs immutable. That information may be reflected in helper methods or metadata if useful, but the policy remains in higher layers.

## Expected Deliverables
- A typed Upstash client abstraction and implementation.
- Request/response DTOs for each supported endpoint.
- Focused error translation that preserves actionable details without leaking secrets.
- Test seams that allow deterministic fake-provider scenarios.

## Expected Areas
- HTTP client implementation
- Auth helpers
- DTOs / provider models
- Provider error types or result wrappers

## Testing Requirements
- Add Reqnroll scenarios that verify:
  - auth headers are constructed correctly
  - successful responses are parsed correctly
  - credential-bearing responses are handled safely
  - provider validation errors and auth failures are surfaced predictably
  - cancellation and non-success status handling behave correctly
- Use fake HTTP handlers or equivalent deterministic doubles rather than live Upstash by default.
- Run the test project and `dotnet build ./Aspire.Hosting.Upstash.Redis.slnx`.

## PR Requirements
The PR must:

- Document the supported endpoints implemented.
- Explain how provider errors are surfaced to higher layers.
- Identify any intentionally unsupported Upstash endpoints and why they stay out of v1.

## Plan Maintenance Requirements
If the real provider contract differs from `0.2`:

- Update `0.2` and all affected downstream task files.
- Do not quietly patch around capability differences in implementation only.

## Done When
- Later deploy tasks can provision and inspect Upstash resources without re-implementing HTTP details.
- The client is fully covered by passing Reqnroll scenarios and safe to build on.
