# Confirm Upstash Management Capability Matrix

Step: `0.2`  
Stage: `0`  
Depends on: `none`  
Can run in parallel with: `0.1`  
Blocks: `1.1`, `2.2`, `2.3`, `3.2`, `3.3`, `4.1`, `4.2`, `4.3`, `5.1`, `6.2`, `6.3`

## Purpose
Build the authoritative Upstash capability map for the package. This task exists to replace assumptions with a precise provider contract covering authentication, creation, lookup, credential retrieval, mutable settings, immutable settings, error conditions, and safety boundaries for repeated deploys.

## Full Description
Investigate the Upstash Redis management API and confirm, with primary-source evidence and prototypes where necessary, the exact behavior the package can rely on. This task must answer all of the following:

1. Authentication contract:
   - confirm the package must use separate `email + API key`
   - confirm whether any single-token or encoded-basic-auth variant is useful or should be intentionally unsupported
2. Account limitations:
   - confirm the developer API works only for native Upstash accounts
   - determine whether this package should fail fast with a tailored error when marketplace-style accounts are detected or inferred
3. Lookup contract:
   - determine whether lookup by explicit database name is safe and unique enough
   - determine whether the package should prefer database id after first discovery when possible
4. Credential retrieval:
   - confirm the exact API response fields that expose the Redis password/token and endpoint details needed to construct a Redis connection string
   - determine whether a separate fetch is required after create
   - determine whether `reset-password` should ever be used automatically or only explicitly never
5. Mutable settings:
   - confirm how to update read regions
   - confirm how to update plan
   - confirm how to update budget
   - confirm how to enable or disable eviction
   - confirm TLS behavior, including whether it is effectively always on or actually mutable in practice
6. Immutable or dangerous settings:
   - confirm whether platform is immutable
   - confirm whether primary region is immutable
   - confirm whether database name should be treated as identity rather than reconciled
   - confirm any settings that are technically mutable but too dangerous for v1
7. State and readiness:
   - confirm whether create/update responses are immediately usable
   - confirm whether polling for `active` state is required after create or reconcile operations
8. Error model:
   - identify important status codes and failure shapes for not found, auth failure, validation failure, conflict, rate limit, and transient provider failure
   - define which failures should be retried and which should fail immediately

The end result must be a concrete matrix that later tasks can implement directly. For each supported option, mark it as one of:

- create-time only
- safely mutable
- readable but not safely mutable
- unsupported in v1

This task must also define the initial package behavior for unresolved or contradictory Upstash documentation. If a real API probe is required to settle a question, say so explicitly and either:

- perform that probe if credentials and permissions are available, or
- document the unresolved point and update the plan so a later implementation task does not silently guess

## Expected Deliverables
- A capability matrix covering auth, lookup, creation, outputs, mutable settings, immutable settings, and failure modes.
- A concrete recommendation for the internal Upstash client contract that `2.2`, `4.1`, `4.2`, and `4.3` can use.
- A documented decision on which provider behaviors are supported in v1 and which are deliberately excluded.
- Updated plan files if any earlier assumptions prove inaccurate.

## Testing Requirements
If this task introduces code:

- Add passing Reqnroll scenarios around any implemented parsing, auth-header generation, or capability flags.
- Add tests for any mapping logic added as part of the investigation.
- Run `dotnet build ./Aspire.Hosting.Upstash.Redis.slnx`.

If this task remains investigative only:

- Run `dotnet build ./Aspire.Hosting.Upstash.Redis.slnx`.
- Include source references and, if used, API probe evidence in the PR.

## PR Requirements
The PR must:

- Include the final capability matrix in readable form.
- Call out any doc contradictions and how they were resolved.
- Spell out the mutable vs immutable setting map for implementation tasks.
- State clearly whether additional plan files were created or edited.

## Plan Maintenance Requirements
If the capability matrix changes the implementation path:

- Update the affected task files immediately.
- Add a new task if a provider limitation creates new work rather than burying that work in notes.
- Do not let unresolved provider ambiguity leak into later implementation tasks.

## Done When
- The package can be implemented against Upstash without hidden provider assumptions.
- Later tasks know exactly what to create, what to reconcile, what to refuse to change, and how to report failures.
- Any unresolved provider gaps are explicitly represented in `plans/`.
