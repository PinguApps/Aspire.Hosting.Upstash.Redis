# Confirm Upstash Management Capability Matrix

Step: `0.2`

## Purpose
Build the authoritative Upstash capability map for the package. This task exists to replace assumptions with a precise provider contract covering authentication, creation, lookup, credential retrieval, mutable settings, immutable settings, error conditions, and safety boundaries for repeated deploys.

## Full Description
Investigate the Upstash Redis management API and confirm, with primary-source evidence and prototypes where necessary, the exact behavior the package can rely on. This task must answer all of the following:

1. Authentication contract:
   - confirm the package must use separate `email + API key`
   - confirm whether any single-token or encoded-basic-auth variant is useful or should be intentionally unsupported
2. Account limitations:
   - confirm the developer API works only for native Upstash accounts
   - determine whether this package should fail fast with a tailored error when marketplace-style accounts are detected or inferred
3. Lookup contract:
   - determine whether lookup by explicit database name is safe and unique enough
   - determine whether the package should prefer database id after first discovery when possible
4. Credential retrieval:
   - confirm the exact API response fields that expose the Redis password/token and endpoint details needed to construct a Redis connection string
   - determine whether a separate fetch is required after create
   - determine whether `reset-password` should ever be used automatically or only explicitly never
5. Mutable settings:
   - confirm how to update read regions
   - confirm how to update plan
   - confirm how to update budget
   - confirm how to enable or disable eviction
   - confirm TLS behavior, including whether it is effectively always on or actually mutable in practice
6. Immutable or dangerous settings:
   - confirm whether platform is immutable
   - confirm whether primary region is immutable
   - confirm whether database name should be treated as identity rather than reconciled
   - confirm any settings that are technically mutable but too dangerous for v1
7. State and readiness:
   - confirm whether create/update responses are immediately usable
   - confirm whether polling for `active` state is required after create or reconcile operations
8. Error model:
   - identify important status codes and failure shapes for not found, auth failure, validation failure, conflict, rate limit, and transient provider failure
   - define which failures should be retried and which should fail immediately

The end result must be a concrete matrix that later tasks can implement directly. For each supported option, mark it as one of:

- create-time only
- safely mutable
- readable but not safely mutable
- unsupported in v1

This task must also define the initial package behavior for unresolved or contradictory Upstash documentation. If a real API probe is required to settle a question, say so explicitly and either:

- perform that probe if credentials and permissions are available, or
- document the unresolved point and update the plan so a later implementation task does not silently guess

### Capability Matrix

| Area | Provider behavior | V1 package behavior |
| --- | --- | --- |
| Management auth | Developer API uses HTTP Basic auth with Upstash account email as username and Management API key as password. | Support separate email and API key inputs only. Construct `Authorization: Basic base64(email:apiKey)` internally. Do not expose the API key to application outputs. |
| Single token auth | The Developer API docs do not define Bearer or single-token auth for Redis management. Bearer tokens are for other Upstash surfaces such as Redis REST/QStash, not management. | Unsupported in v1. Reject or avoid any API shape that implies a single management token. |
| Account type | Developer API is only available to native Upstash accounts. Third-party platform accounts such as Vercel or Fly.io are not supported. | Fail fast with a tailored error when auth/permission failure likely indicates a marketplace-style account. The message should tell the user to use a native Upstash account and Management API key. |
| Lookup | Provider supports list all databases and get by database id. No documented get-by-name endpoint exists. List responses include `database_id` and `database_name`. | Lookup by explicit database name through `GET /redis/databases`, requiring exactly one case-sensitive match after applying the package's chosen comparison rule. Persist/prefer provider id after discovery if Aspire deployment state gives a safe place to store it; otherwise repeat deterministic name lookup. |
| Stable identity | Provider id is the only unique database identity in the management API. Database name can be renamed through a provider endpoint. | Treat configured remote database name as desired identity selector. Do not auto-rename. If a stored id exists, prefer id and verify the remote name still matches the explicit configured name. |
| Create | `POST /redis/database` creates a global Redis database with required `database_name`, `platform`, and `primary_region`; optional `read_regions`, `plan`, `budget`, `eviction`, and `tls`. Response schema is `Database`. | Create-time supported options: database name, platform, primary region, read regions, plan, budget, eviction. Send `tls: true` only if the public API still exposes TLS; reject `false`. |
| Credential retrieval | Current `GET /redis/database/{id}` docs define `credentials=hide` to remove credentials, but the current rendered schema omits `password`/REST tokens. Terraform docs and older API reference identify `password`, `rest_token`, and `read_only_rest_token` as database attributes. | The client must call `GET /redis/database/{id}` without `credentials=hide` after create/adopt/reconcile and require `password`, `endpoint`, `port`, and `tls` for outputs. If `password` is absent, fail with an explicit provider-contract error instead of guessing or resetting. REST tokens are readable but not used for v1 Redis TCP outputs. |
| Reset password | `POST /redis/reset-password/{id}` exists and rotates credentials. | Never call automatically in v1. It would invalidate existing consumers. Leave explicit password rotation out of scope. |
| Read regions | `POST /redis/update-regions/{id}` accepts `read_regions` and returns `"OK"`. | Safely mutable for explicit desired values. Re-fetch after update. |
| Plan | `POST /redis/change-plan/{id}` accepts `plan_name`; docs also include `auto_upgrade` and `prod_pack_enabled`. Upgrading may cause brief downtime per Upstash upgrade docs. | Safely mutable only for explicit `plan` changes. Do not manage `auto_upgrade` or `prod_pack_enabled` in v1. Re-fetch after update and fail if provider does not converge. |
| Budget | `PATCH /redis/update-budget/{id}` accepts `budget` and returns `"OK"`. Budget applies to pay-as-you-go behavior. | Safely mutable for explicit desired values. Validate/document plan compatibility in domain/reconcile work. |
| Eviction | Create supports `eviction`; dedicated enable/disable endpoints exist and return `"OK"`. | Safely mutable for explicit desired values. Re-fetch after update. |
| TLS | Create docs include `tls`; an `enable-tls` endpoint exists. Redis security docs say TLS is always enabled, and Terraform docs describe TLS as encrypted-in-transit/read-only in practice. No disable endpoint exists. | Readable but not safely mutable. V1 should require/assume TLS enabled and fail if a user explicitly asks for `false` or if provider state reports `tls=false`. Do not call `enable-tls` automatically unless a later live probe proves it is needed and safe. |
| Platform | Create requires `platform`; read responses do not reliably expose it as a direct field. No update platform endpoint exists. | Create-time only. Persist intended platform in package state if possible; otherwise immutable drift can only be detected when provider state exposes enough evidence. |
| Primary region | Create requires `primary_region`; read responses expose `primary_region`. No update primary-region endpoint exists. | Create-time only. Existing database mismatch fails immutable drift detection. |
| Database name | Rename endpoint exists. | Treat as identity, not a reconciled setting. Do not rename in v1. A changed configured remote name means a different target selection. |
| Delete | Delete endpoint exists. | Unsupported in v1. Never auto-delete remote Upstash databases. |
| Auto upgrade / prod pack / consistency / backups / ACL / IP allowlist / private networking / team move | Provider exposes some endpoints/fields, but they are outside the planned v1 Redis publishing contract or have security/billing blast radius. | Unsupported in v1. Read only when useful for diagnostics; do not mutate. |
| State/readiness | Responses expose `state` (`active`, `suspended`, `passive`) and `modifying_state`. Create examples return `active`, but no primary docs guarantee immediate readiness after every operation. | Poll `GET /redis/database/{id}` after create and after mutations until `state == active` and `modifying_state` is empty, with bounded timeout/backoff. Treat `suspended`/`passive` as not ready and fail unless later evidence defines a safe wait path. |
| Endpoint shape | Docs say `endpoint` may be a slug or full hostname. Older examples and Terraform docs show a full `*.upstash.io` hostname. | Client/output code must normalize endpoint safely: use full host when provided; if only a slug is returned, construct/validate the expected Upstash host only if provider docs or live probe confirms the suffix rule. Otherwise fail clearly rather than emitting a broken connection string. |
| Error model | Status docs list 400, 401, 403, 404, 405, 406, 429, 500, and 503. Probe confirmed 401 can be empty or `{"error":"Unauthorized"}`. | Fail immediately for 400/401/403/404/405/406 except 404 may drive not-found ownership behavior. Retry bounded for 429, 500, 503, and transport timeouts. Preserve provider error text when present, redacting secrets. |

### Internal Client Contract Recommendation

The `2.2` client should expose a narrow Redis management contract:

- `ListDatabasesAsync`
- `GetDatabaseAsync(id, includeCredentials: true)`
- `CreateDatabaseAsync(request)`
- `UpdateReadRegionsAsync(id, readRegions)`
- `ChangePlanAsync(id, planName)`
- `UpdateBudgetAsync(id, budget)`
- `SetEvictionAsync(id, enabled)`

The client should also provide shared response models with these v1 fields: `database_id`, `database_name`, `endpoint`, `port`, `password`, `tls`, `state`, `modifying_state`, `primary_region`, `read_regions`, `type`, `budget`, `eviction`, `customer_id`, and optional raw provider fields needed for diagnostics. `reset-password`, `rename`, `delete`, `enable-tls`, team move, backup, autoscaling, prod pack, ACL, and networking endpoints should stay unimplemented unless a later task explicitly expands v1 scope.

### Contradictions and Resolution

- TLS is contradictory: create/enable API pages imply configurability, while Redis security says TLS is always enabled. Resolve by treating TLS as always-on/read-only for v1.
- Credential fields are contradictory: current OpenAPI text has a `credentials=hide` query parameter but omits credential fields from the displayed `Database` schema, while Terraform docs list `password` and REST tokens. Resolve by requiring a post-create/detail fetch and failing clearly if the live provider omits `password`.
- Endpoint shape is underspecified: current docs allow slug or hostname. Resolve by making endpoint normalization explicit and fail-fast unless the returned value is a full host or a confirmed slug-to-host rule exists.

### Downstream Plan Impact

Updated downstream plan files must implement this matrix as follows:

- `2.2`: implement only the narrow client contract listed above.
- `2.3`: model TLS as required-on/read-only, not a mutable user option.
- `3.3`: use name lookup plus provider id verification/persistence when available.
- `4.1`: follow create with detail fetch and readiness polling.
- `4.2`: reconcile read regions, plan, budget, and eviction only.
- `4.3`: fail immutable drift for primary region, database name identity mismatch, TLS false, platform where detectable, and unsupported settings.
- `5.1`/`5.2`: build outputs only from credential-bearing detail responses.
