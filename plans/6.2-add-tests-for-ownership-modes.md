# Add Tests for Ownership Modes

Step: `6.2`  
Stage: `6`  
Depends on: `3.2`, `3.3`, `4.1`, `5.3`  
Can run in parallel with: `6.1`, `6.3`  
Blocks: `7.3`

## Purpose
Prove the central provisioning contract end-to-end for create-only, existing-only, and create-or-adopt behavior.

## Full Description
Expand the Reqnroll suite so ownership-mode behavior is covered as a first-class scenario matrix rather than only through unit-level behavior checks from earlier tasks. This is the hardening pass for the main deployment contract.

The test matrix should cover:

- create-only create success
- create-only existing collision failure
- existing-only success
- existing-only missing-resource failure
- create-or-adopt create path
- create-or-adopt adopt path
- repeated deploy behavior under each relevant mode
- name/identity edge cases discovered in `3.3`

Where earlier tasks already covered parts of this behavior, consolidate and deepen the scenarios instead of duplicating shallow tests.

## Expected Deliverables
- End-to-end Reqnroll coverage for ownership-mode behavior.
- Any focused implementation fixes required to close discovered gaps.

## Testing Requirements
- Add passing scenarios for the full ownership matrix.
- Ensure failure messages are asserted, not just exception types.
- Run the full test suite and `dotnet build ./Aspire.Hosting.Upstash.Redis.slnx`.

## PR Requirements
The PR must:

- Show the ownership matrix now covered.
- Call out any behavior change required to make the tests pass.

## Plan Maintenance Requirements
If ownership behavior turns out to need another implementation task:

- Add or update the relevant files in `plans/` in the same PR.

## Done When
- Ownership semantics are comprehensively covered by passing Reqnroll scenarios and are ready for release confidence.
