## Purpose
Implement the actual Upstash Redis database creation path used when ownership resolution says a new remote database must be provisioned.

## Full Description
Build the deploy-time creation flow for Upstash Redis using the typed provider client and the resolved configuration model. This task must turn a validated deployment request into a real remote database and fetch the information later tasks need for outputs.

The create flow must:

1. Build the provider create request from the package domain model.
2. Use only the supported v1 config surface:
   - database name
   - platform
   - primary region
   - read regions
   - plan
   - budget
   - eviction
   - TLS as required-on only; reject `false` clearly, and do not treat TLS as mutable
3. Consume the `3.2` ownership-resolution result and run only when it selects the create path.
4. Confirm the created database is ready enough for downstream reconcile and output steps.
5. Retrieve all connection details required for app-facing Redis outputs without leaking secrets in logs.

The `0.2` capability matrix requires a follow-up `GET /redis/database/{id}` without `credentials=hide` after creation. Use that response to obtain `password`, endpoint, port, TLS state, and current database settings. If `password` is missing, fail with a provider-contract error; do not call `reset-password`. Poll readiness with bounded backoff until `state == active` and `modifying_state` is empty.

This task must also define the observable deployment behavior around creation:

- what progress is reported
- what high-level success is logged
- what failure message is shown when creation fails

## Implementation Result
State: `Completed`

Task `4.1` added the internal `UpstashRedisCreateFlow` and wired the deploy pipeline through deploy-time resolution, remote identity verification, ownership resolution, create execution, and remote identity persistence.

The create lifecycle is:

1. Resolve deployment inputs and provider-domain values.
2. Load cached remote identity state, verify it through the remote identity resolver, and use the verified database for ownership resolution.
3. Resolve ownership by explicit database name.
4. Run create only when ownership selects `Create`; adopt results are passed through without a create call.
5. Build `POST /redis/database` from the resolved v1 surface: database name, platform, primary region, read regions, plan, budget, eviction, and TLS.
6. Send TLS as `true`; `false` remains rejected by the option model and the create flow.
7. Require explicit platform and primary region before creating because Upstash requires both values and v1 does not define package defaults.
8. Poll `GET /redis/database/{id}` through `WaitUntilReadyAsync` until `state == active` and `modifying_state` is empty.
9. Return the credential-bearing ready database details for later reconcile/output tasks and persist the resolved remote identity state for repeated deploy drift protection.

Credential retrieval uses the existing management client behavior: readiness polling fetches `GET /redis/database/{id}` without `credentials=hide`, and the create flow requires the ready detail response to contain `password`, endpoint, port, and TLS enabled state. If any required connection field is missing or unusable, deployment fails with `UpstashRedisProviderFailureKind.ProviderContract` and does not call `reset-password`.

Observable behavior for this task is intentionally high-level and secret-safe: the pipeline logs when it starts resolving the configured database, logs whether it created a new database or used an existing one, wraps provider create failures with `Failed to create Upstash Redis database '<name>': ...`, leaves readiness timeout/failure messages to the management client, and never logs passwords or management credentials. Task `5.3` still owns richer progress reporting and diagnostics.

Reqnroll coverage was added in `Features/DeployReconcileOutputs/CreateFlow.feature` for successful create, payload mapping, credential retrieval after create, readiness waiting, returned remote identity state, provider create failure wording, missing connection-field provider-contract failures, and adopt no-op behavior.
